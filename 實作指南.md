## 一、資料來源 API 呼叫範例

### 找標案 — g0v 標案 API（已實測可用）

Base URL：`https://pcc-api.openfun.app`（資料源：政府電子採購網；由 g0v／OpenFun 維護）

| 用途 | 呼叫 | 回傳 |
|---|---|---|
| 關鍵字搜尋 | `GET /api/searchbytitle?query=冷氣&page=1` | `{query, page, total_records, total_pages, records:[…]}` |
| 依日期列出 | `GET /api/listbydate?date=20260610` | `{records:[…]}` |
| 單一標案明細 | `GET /api/tender?unit_id={unit_id}&job_number={job_number}` | 完整標案 JSON |

每筆 record 結構：

```json
{
  "date": 20260610,
  "filename": "ttd-71243636",
  "job_number": "YY1150017-01",
  "unit_id": "A.21.100.33",
  "unit_name": "衛生福利部桃園醫院",
  "brief": { "type": "公開取得報價單或企劃書", "title": "…", "category": "財物類481-醫療…" },
  "tender_api_url": "https://pcc-api.openfun.app/api/tender?unit_id=…&job_number=…",
  "url": "https://web.pcc.gov.tw/…"
}
```

💡 每筆 record 已附 `tender_api_url`，直接 GET 它就拿到明細，不用自己組。
⚠️ **禮貌限速**（每秒 1–2 次）、標註來源、本機快取——這是公民團體的服務，別大量打（它曾因 AI 爬蟲過載上 Cloudflare）。

### 決標行情 — 政府電子採購網「決標開放資料」
官方 OpenData 檔案下載 → 定期匯入本地 DB 做統計（金額／底價／決標比）。

### 找補助清單 — 文化部開放資料（XML）
`GET https://opendata.culture.tw/frontsite/openData/detail?datasetId=351`（獎補助-報名資訊）
欄位：`Year / OrganName / UnitName / PointName(計畫名稱) / PublishDate / Property(獎勵或補助) / URL`
其餘補助多無開放 API，請見主 README 的「官方查詢入口」逐站查。

## 二、資料模型草圖（本地 SQLite / PostgreSQL）

| 表 | 主要欄位 | 用途 |
|---|---|---|
| `tenders` | filename(PK), date, unit_id, unit_name, title, category, budget, deadline, raw_json | 標案快取 |
| `awards` | id(PK), org, case_no, name, budget, base_price, award_price, award_date, winners | 決標（OpenData 匯入）|
| `bidders` | award_id(FK), name, won | 投標廠商（分析用）|
| `grants` | sid(PK), name, org, region, target, intro, apply, amount, url | 補助清單 |
| `profiles` | id(PK), name, description | 公司檔案（AI 比對用）|
| `tracked` | pk(PK), payload, note, added_at, status | 我的追蹤 |

## 三、AI prompt 範本

### 標案適合度分析
```
你是政府標案投標顧問。以下是一個標案的公告全文，以及一家公司的描述。
請判斷這家公司投這個案子的「適合度」，並嚴格遵守：
- 每個門檻／風險判斷，都要附「公告原文片段」當依據（沒有原文依據的不要寫）
- 不確定的標「待確認」，不要臆測
輸出：總評（適合／有條件適合／不適合）＋ 逐項資格檢核（附原文）＋ 要準備的文件 ＋ 風險。

【標案公告】{tender_full_text}
【公司描述】{company_profile}
```

### 補助資格配對（兩段式的第 2 段）
```
你是補助申請顧問。以下是「一批補助計畫的資格條文」與「一家公司的描述」。
逐筆判斷公司是否符合，分成 符合／可能符合／不符合，每筆附一句理由（引用資格條文關鍵句）。
不確定就歸「可能符合」並說明缺什麼資訊。

【公司描述】{company_profile}
【補助清單】{grants_with_eligibility}
```
> 第 1 段「規則預篩」用**程式**做：先用地區、對象、關鍵字過濾，把明顯不符的刷掉，再把剩下的丟給 AI → 省 token、又快。

## 四、建造步驟（P0 → P5）

- **P0 地基**：建專案、本地 DB、一個能跑的後端 ＋ 空白前端
- **P1 找標案**：接 g0v API `searchbytitle` → 存 `tenders` → 前端搜尋／篩選／卡片
- **P2 決標行情**：下載決標 OpenData → 匯入 `awards`／`bidders` → 統計
- **P3 AI 適合度**：點標案 → 抓 `tender_api_url` 明細 → 丟上面的 prompt → 顯示結果（附原文）
- **P4 找補助**：匯入文化部 351 → 建公司檔案 → 兩段式配對（規則預篩 ＋ AI 判讀）
- **P5 分析／追蹤**：廠商／機關分析（從 `awards`／`bidders` 聚合）、追蹤清單 ＋ 截止提醒、可選本地 LLM

## 五、合規提醒（做之前一定要看）

- g0v API／開放資料：**限速、標來源、本機快取**；別架成大量消耗別人服務的公開站
- 補助資格詳情無開放資料 → **link-out 官方頁，別爬 robots 禁止的站**（如 grants.moc.gov.tw）
- **自部署＝個人使用最穩**；判定結果以官方公告為準
